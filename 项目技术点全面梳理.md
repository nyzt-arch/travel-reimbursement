## 一、项目架构总览

### 1.1 整体架构模式

```
┌──────────────────────┐       HTTP/REST        ┌──────────────────────┐
│    reimbursement-front │ ◄──────────────────► │  reimbursement-back   │
│    (Vue 3 + Vite)     │    JSON 数据交互       │  (Spring Boot 4.1)    │
│    端口: 5173          │                       │  端口: 8080            │
└──────────────────────┘                        └──────────┬───────────┘
                                                           │
                                                           │ JDBC
                                                           ▼
                                                  ┌──────────────────┐
                                                  │     MySQL 8.x     │
                                                  │  travel-reimbursement │
                                                  └──────────────────┘
```

**架构特点：**
- **前后端分离架构**：前端通过 Vite 开发服务器运行在 5173 端口，后端 Spring Boot 运行在 8080 端口
- **Vite 代理解决跨域**：前端 `/api` 前缀请求被代理到后端，`changeOrigin: true` + `rewrite` 去掉 `/api` 前缀
- **前后端独立部署**：开发环境和生产环境均可独立构建和部署

### 1.2 分层架构（后端）

```
Controller 层 (REST API 入口)
    │
    ▼
Service 层 (业务逻辑)
    │
    ▼
Mapper 层 (数据访问 - MyBatis-Plus BaseMapper)
    │
    ▼
Entity 层 (数据实体 - 与数据库表映射)
```

**数据传输对象流转：**

```
前端请求 → DTO (Data Transfer Object) → Service 处理 → Entity → 数据库
数据库 → Entity → Service 组装 → VO (View Object) → 前端响应
```

### 2.2 MyBatis-Plus 3.5.16

这是本项目最为核心的技术栈之一，需要深入理解。

**2.2.1 BaseMapper — 零 SQL 实现 CRUD**

```java
@Mapper
public interface FkReimMainMapper extends BaseMapper<FkReimMain> {
    // 无需写任何方法，BaseMapper 已提供：
    // insert(), deleteById(), updateById(), selectById()
    // selectList(wrapper), selectPage(page, wrapper)
}
```

**BaseMapper 提供的内置方法（面试重点）：**

| 方法 | 说明 |
|------|------|
| `insert(entity)` | 插入一条记录 |
| `deleteById(id)` | 根据ID删除 |
| `updateById(entity)` | 根据ID更新 |
| `selectById(id)` | 根据ID查询 |
| `selectList(wrapper)` | 条件查询列表 |
| `selectPage(page, wrapper)` | 分页查询 |
| `selectCount(wrapper)` | 条件计数 |
| `saveOrUpdate(entity)` | 有ID则更新，无则插入 |

**2.2.2 LambdaQueryWrapper — 类型安全的动态查询**

```java
LambdaQueryWrapper<FkReimMain> wrapper = new LambdaQueryWrapper<>();
// 模糊查询（LIKE）
wrapper.like(FkReimMain::getReimNo, queryDTO.getReimNo());
// 精确匹配（=）
wrapper.eq(FkReimMain::getReimCompanyId, queryDTO.getCompanyId());
// 排序
wrapper.orderByDesc(FkReimMain::getCreationTime);
```

**优势（面试需强调）：**
- **类型安全**：使用 Lambda 方法引用（如 `FkReimMain::getReimNo`），字段名变更时编译期即可发现，避免字符串拼写错误
- **防 SQL 注入**：参数化查询，MyBatis-Plus 底层使用 `PreparedStatement`
- **链式调用**：代码可读性高，动态条件拼接简洁

**2.2.3 分页插件 — 物理分页**

```java
@Configuration
@MapperScan("com.example.reimbursementback.mapper")
public class MyBatisPlusConfig {
    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor() {
        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
        // 添加分页插件，指定数据库类型为 MySQL
        interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
        return interceptor;
    }
}
```

**工作原理：**
1. 前端传入 `pageNum`（页码）和 `pageSize`（每页条数）
2. 构造 `Page<FkReimMain> page = new Page<>(pageNum, pageSize)`
3. `PaginationInnerInterceptor` 拦截 SQL，自动拼接 `LIMIT offset, size`
4. 返回的 `Page` 对象包含 `records`（数据列表）、`total`（总记录数）、`current`（当前页）、`size`（每页大小）

**面试要点：**
- 必须注册 `PaginationInnerInterceptor` 才能实现物理分页，否则 MyBatis-Plus 会进行内存分页（全量查询再截取，性能极差）
- `DbType.MYSQL` 参数告诉插件生成 MySQL 兼容的 LIMIT 语法
- `@MapperScan` 注解一次扫描整个 Mapper 包，避免在每个 Mapper 接口上单独加 `@Mapper`

**2.2.4 主键策略 — 雪花算法（Snowflake）**

```java
@TableId(type = IdType.ASSIGN_ID)
private String id;
```

**技术点：**
- `IdType.ASSIGN_ID`：使用雪花算法生成全局唯一的 64 位 Long 型 ID（转为 String 存储）
- **结构**：1位符号位 + 41位时间戳 + 10位工作机器ID + 12位序列号
- **优势**：分布式环境下不依赖数据库自增，保证全局唯一，趋势递增
- 对于基础数据表（`BaseCompany` 等），使用 `IdType.INPUT`（手动输入/已有ID）

### 2.6 统一响应格式 — Result 类

```java
@Data
public class Result<T> {
    private Integer code; // 200=成功, 500=失败
    private String msg;   // 提示信息
    private T data;       // 泛型数据

    public static <T> Result<T> success(T data) { ... }
    public static <T> Result<T> error(String msg) { ... }
}
```

**设计意图：**
- 前端 Axios 拦截器根据 `code === 200` 判断请求是否成功
- 统一的响应结构使前端可以写一个统一拦截器处理所有接口返回
- 泛型 `T` 使不同类型的业务数据都能复用同一个 Result 结构

### 2.7 全局异常处理

```java
@RestControllerAdvice  // = @ControllerAdvice + @ResponseBody
public class GlobalExceptionHandler {

    @ExceptionHandler(RuntimeException.class)
    public Result<Void> handleRuntimeException(RuntimeException e) {
        return Result.error(e.getMessage());
    }
}
```

**工作原理（面试重点）：**
- `@RestControllerAdvice` 全局拦截所有 Controller 抛出的异常
- `@ExceptionHandler(RuntimeException.class)` 指定捕获的异常类型
- **关键设计**：将异常转换为 HTTP 200 + code:500 的响应，而不是返回 HTTP 500 状态码
  - 好处：前端可以直接从 `res.code` 判断业务错误，而不需要额外处理 HTTP 状态码
  - 如果不这样做，前端 Axios 在遇到 HTTP 500 时会进入 error 回调，只能看到 "Request failed with status code 500"，丢失了具体的错误信息

**业务层抛出异常的方式：**
```java
if (!StringUtils.hasText(saveDTO.getTitle())) {
    throw new RuntimeException("报销单标题不能为空！");
}
```

### 2.8 事务管理

```java
@Override
@Transactional(rollbackFor = Exception.class)
public void submitReimbursement(ReimbursementSaveDTO saveDTO) {
    // 业务校验...
    saveOrUpdateMain(saveDTO, "1");
}
```

**面试要点：**
- `@Transactional(rollbackFor = Exception.class)` 确保任何异常（包括 checked exception）都会触发回滚
- 默认情况下 Spring 只对 RuntimeException 和 Error 进行回滚
- 在报销单场景中，主表 + 多个子表（行程、补助、分摊）需要保证数据一致性，事务至关重要

### 2.9 synchronized 并发控制

```java
@PostMapping("/save")
public Result<String> save(@RequestBody ReimbursementSaveDTO saveDTO) {
    synchronized (ReimbursementController.class) {
        String id = fkReimMainService.saveDraft(saveDTO);
        return Result.success(id);
    }
}
```

**作用：**
- 使用类级别的 `synchronized` 锁防止并发请求导致**报销单号重复**
- 单号生成逻辑 `"RCBX" + date + String.format("%03d", count + 1)` 在高并发下可能出现重复
- 实际生产环境中应该使用数据库唯一约束 + 重试机制，或使用 Redis 分布式锁

### 2.10 Java 8+ Time API

```java
// 时间格式化器
private static final DateTimeFormatter DATETIME_FORMATTER =
    DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss");
private static final DateTimeFormatter DATE_FORMATTER =
    DateTimeFormatter.ofPattern("yyyy-MM-dd");

// LocalDateTime 转字符串
entity.getCreationTime().format(DATETIME_FORMATTER)

// 字符串转 LocalDate
LocalDate.parse(tripDTO.getDepartDate(), DATE_FORMATTER)
```

**为何不用 SimpleDateFormat：**
- `DateTimeFormatter` 是线程安全的（不可变对象）
- `LocalDate` / `LocalDateTime` 提供了更清晰的日期时间 API
- 避免了旧 API（`java.util.Date`、`java.sql.Date`）的线程安全和设计问题

### 2.12 树形数据结构构建

```java
// BaseDataServiceImpl.getBusinessTypeTree()
for (BaseBusinessType type : all) {
    if ("none".equalsIgnoreCase(type.getSuperiorId())) {
        roots.add(type);
```

**技术点：**

- 时间复杂度 O(n)，空间复杂度 O(n)

### 2.13 冗余字段策略

项目中主表和子表都采用了**冗余存储**策略：

```
主表 fk_reim_main 同时存储：
  - reimDepartmentId (ID) + reimDepartmentNo (编号) + reimDepartmentName (名称)

子表 fk_reim_trip 同时存储：
  - personId (ID) + personNo (工号) + personName (姓名)
  - departureCityNo (编号) + departureCityName (名称)
```

**优势：**
- 查询时不需要 JOIN 多张表，提升查询性能
- 历史数据不受基础数据变更影响（例如员工离职后，历史报销单仍能看到当时的信息）
- **以空间换时间**

**代价：**
- 数据冗余增加存储空间
- 基础数据变更时需要同步更新，或业务上选择不更新（保留历史快照）

### 2.14 报销单号自动生成

```java
// 格式: RCBX + yyyyMMdd + 001（序号）
String dateStr = LocalDate.now().format(DateTimeFormatter.ofPattern("yyyyMMdd"));
String prefix = "RCBX" + dateStr;
Long todayCount = this.count(
    new LambdaQueryWrapper<FkReimMain>()
        .like(FkReimMain::getReimNo, prefix + "%")
);
String reimNo = prefix + String.format("%03d", todayCount + 1);
```

**技术点：**
- `%03d` 格式化确保序号始终为三位数（001, 002, ..., 999）
- `String.format()` 是 Java 标准库的格式化方法
- 用 `like` 查询当天已有单号数量，+1 作为新单号

### 3.3 Vue Router 5.1

```typescript
const router = createRouter({
```

```typescript
  state: () => ({

    departments: [] as Department[],
    employees: [] as Employee[],

    getCompanyNameById: (state) => (id: string) => {
      return state.companies.find((c) => c.id === id)?.name || ''
    },
    getEmployeeNameById: (state) => (id: string) => {
      const emp = state.employees.find((e) => e.id === id)
      return emp ? `${emp.name}[${emp.no}]` : ''

    // 树形搜索（递归查找）

      return (id: string): string => {

          for (const node of nodes) {
            if (node.id === id) return node.name
            if (node.children) {
              const res = findInTree(node.children)
              if (res) return res
            }
          }
          return ''
        }
        return findInTree(state.businessTypes)
      }
    },
  },
  actions: {
    async loadAllBaseData() {

        await Promise.all([               // 并发加载 6 个接口
          baseDataApi.getCompanies(),
          baseDataApi.getDepartments(),
          // ... 4 more
```

**Pinia 设计亮点：**

```typescript
export function getDatesBetween(startDateStr: string, endDateStr: string) {
  const dates: { date: string; weekday: string }[] = []
  let current = new Date(startDateStr)
  const end = new Date(endDateStr)
  const weekdayNames = ['星期日', '星期一', ...]

  while (current <= end) {
    dates.push({
      date: `${year}-${month}-${day}`,
      weekday: weekdayNames[current.getDay()]
    })
    current.setDate(current.getDate() + 1)
  }
  return dates
}
```

### 5.2 路径命名规范

- 后端路径与前端 Vite 代理配合：前端请求 `/api/reimbursement/list` → 代理去掉 `/api` → 后端收到 `/reimbursement/list`
- 基础数据路径用下划线 `base_data`
- 资源名用名词复数或单数
- 层级关系用路径表达（如 `/detail/{id}`）

### 5.3 GET vs POST 选择

- **GET**：查询操作，参数通过 URL Query String 传递（如列表查询的筛选条件）
  ```java
  @GetMapping("/list")
  public Result<Page<ReimbursementListVO>> list(ReimbursementQueryDTO queryDTO) { }
  ```
  Spring MVC 自动将 Query 参数绑定到 DTO 对象（无需 `@RequestParam` 逐字段标注）

- **POST**：创建/修改操作，参数通过 Request Body（JSON）传递
  ```java
  @PostMapping("/save")
  public Result<String> save(@RequestBody ReimbursementSaveDTO saveDTO) { }
  ```
  `@RequestBody` 将 JSON 反序列化为 Java 对象

## 六、核心业务流程

### 6.1 报销单状态流转

```
┌─────────┐    提交(submit)    ┌──────────┐
│  草 稿   │ ──────────────► │  已完成   │
│ (status=0)│                  │ (status=1) │
└─────────┘                  └──────────┘
     │                              │
     │      作废(cancel)             │    作废(cancel)
     └──────────────────────────────┘
                   │
                   ▼
            ┌──────────┐
            │  已作废   │
            │ (status=2) │
            └──────────┘
```

**状态说明：**
- **草稿 (0)**：可编辑、可删除、可提交、可作废
- **已完成 (1)**：可在列表页直接提交（等同于再次提交）、可作废、不可编辑
- **已作废 (2)**：只读，不可任何操作

### 6.2 保存草稿流程

```
1. 前端构造 ReimbursementSaveDTO（含主表信息 + trips + subsidies + allocations）
2. POST /reimbursement/save
3. 后端 saveOrUpdateMain(saveDTO, "0")
   ├── 判断新增 or 修改（通过 id 前缀 NEW_/COPY_/T_/R）
   ├── 新增 → 生成雪花 ID + 自动生成报销单号 (RCBX + yyyyMMdd + 序号)
   ├── 冗余字段补全（查字典表填充名称/编号）
   ├── 保存主表 → saveOrUpdate()
   ├── 修改 → 清空旧子表数据
   ├── 批量保存行程、补助日历、费用分摊
   └── 返回 reimId
4. 前端更新 model.id → 后续自动保存生效
```

### 6.3 提交报销单流程

```
1. 前端先进行客户端校验（validateReimbursement）
   ├── 必填字段不为空
   ├── 同一出行人行程无日期重叠
   ├── 分摊比例之和 = 100%
   └── 分摊金额之和 = 补助总金额

2. POST /reimbursement/submit（经过 synchronized 并发控制）

3. 后端再次进行服务端校验
   ├── 标题、报销人、部门、公司、业务类型必填
   ├── 行程明细不能为空
   ├── 分摊比例之和 = 100%（BigDecimal 精确比较）
   └── 分摊金额 = 补助总金额

4. 校验通过 → saveOrUpdateMain(saveDTO, "1")
   └── status = "1"（已完成）
```

### 6.4 删除报销单（级联删除）

```java
@Transactional(rollbackFor = Exception.class)
public void deleteReimbursement(String id) {
    tripService.deleteTripsByReimId(id);         // 删除行程
    subsidyService.deleteSubsidiesByReimId(id);   // 删除补助（含日历）
    costAllocationService.deleteAllocationsByReimId(id); // 删除分摊
    this.removeById(id);                          // 删除主表
}
```

**技术点：**
- `@Transactional` 保证原子性：全部删除成功或全部回滚
- 删除顺序：先删子表，后删主表（符合外键约束要求）
- 补助删除内部会级联清空日历明细

## 十、常见面试问题及回答要点

### Q1: 为什么选择 MyBatis-Plus 而不是 MyBatis 或 JPA？

**回答要点：**
- **MyBatis-Plus 在 MyBatis 基础上的增强**：内置 CRUD、分页、条件构造器，减少大量重复 SQL
- **LambdaQueryWrapper 类型安全**：编译期检查字段名，避免字符串拼写错误
- **比 JPA 更灵活**：JPA 的 HQL 自动生成难以优化复杂查询，MyBatis-Plus 可以手写 SQL
- **学习成本低**：团队熟悉 MyBatis 生态

### Q2: 项目中的分页是怎么实现的？

**回答要点：**
1. 注册 `PaginationInnerInterceptor(DbType.MYSQL)` 分页插件
2. 前端传入 `pageNum` 和 `pageSize`
3. 后端构造 `Page<Entity>(pageNum, pageSize)` + `LambdaQueryWrapper` 条件
4. 插件拦截 SQL 自动拼接 `LIMIT offset, size`（物理分页）
5. 返回 `Page` 对象包含 `records`、`total`、`current`、`size`
6. 前端 Pagination 组件展示页码、支持快速跳转

### Q3: 如何保证报销单号的唯一性？

**回答要点：**
- 单号格式：`RCBX + yyyyMMdd + 001`（001 为当天序号）
- 使用 `synchronized (ReimbursementController.class)` 保证同一时刻只有一个线程生成单号
- 先 `count` 当天已有单号数，再 +1
- **局限性**：单机锁，分布式环境中应使用数据库唯一约束 + 重试、Redis 分布式锁或号段模式

### Q4: 为什么要做前后端双重校验？

**回答要点：**
- **前端校验**：提升用户体验，即时反馈，减少无效请求
- **后端校验**：保证数据安全性，前端校验可以被绕过（修改 JS、直接 curl 请求）
- **关键数据**：分摊比例 = 100%、分摊金额 = 补助总金额、必填字段等，必须后端再次校验
- 这是一个安全最佳实践

### Q5: 项目中如何处理跨域问题？

**回答要点：**
- **开发环境**：使用 Vite 的 `server.proxy` 配置代理
  ```ts
  proxy: {
    '/api': {
      target: 'http://localhost:8080',
      changeOrigin: true,
      rewrite: (path) => path.replace(/^\/api/, '')
    }
  }
  ```
- 前端请求 `/api/reimbursement/list` → 代理转发到 `http://localhost:8080/reimbursement/list`
- **生产环境**：可使用 Nginx 反向代理，或 Spring Boot 配置 CORS（`@CrossOrigin` 或 `CorsFilter`）

### Q6: 为什么选择 Pinia 而不是 Vuex？

**回答要点：**
- Pinia 是 Vuex 5 的官方替代品，Vue 官方推荐
- **TypeScript 支持更好**：无需复杂的类型标注
- **API 更简洁**：没有 mutations，直接用 actions 修改 state
- **模块化更自然**：每个 store 是独立的 `defineStore` 调用，无需 `modules` 嵌套
- **更好的 DevTools 支持**

### Q7: 行程和补助之间的联动是如何实现的？

**回答要点：**
- 行程数据变化时（新增/修改/删除），`handleUpdateTrips` 函数自动更新补助数据
- 根据行程的到达城市查询城市等级（一线/二线/三线）
- 根据城市等级确定补助标准（餐补 100/80/50，交通和通讯统一 40）
- 根据出发日期和到达日期计算补助天数
- 自动生成补助日历明细（每一天的餐补/交通补/通讯补）
- 重新计算总金额并触发分摊联动

### Q8: 为什么实体类同时存储 ID 和编号/名称？

**回答要点（冗余字段策略）：**
- **查询性能**：不需要 JOIN 多张表获取名称
- **历史快照**：基础数据（如公司名称、员工姓名）变更后，历史报销单仍显示当时的准确信息
- **灵活显示**：前端可以根据需要选择显示 ID、编号或名称
- 代价是存储冗余，但报销单数据量通常不大，冗余开销可接受

### Q9: 项目中 `@Transactional(rollbackFor = Exception.class)` 的作用？

**回答要点：**
- Spring 默认只对 `RuntimeException` 和 `Error` 回滚
- 加上 `rollbackFor = Exception.class` 后，任何 Exception（包括 checked exception）都会触发回滚
- 报销单保存涉及主表 + 多个子表，需要保证数据一致性
- 级联删除也需要事务保证：要么全部删除成功，要么全部不动

### Q10: `synchronized (ReimbursementController.class)` 的含义和局限性？

**回答要点：**
- 类级别的锁，同一时刻只允许一个线程进入该 Controller 的任何 synchronized 方法
- 目的是防止并发请求导致报销单号重复
- **局限性**：
  - 只适用于单机部署（JVM 级别锁）
  - 分布式/集群环境需要分布式锁（Redis、ZooKeeper）
  - 性能瓶颈：所有用户的保存/提交操作串行化

### Q11: `v-if` 和 `v-show` 的区别？项目中分别在哪里使用？

**回答要点：**
- `v-if`：条件为假时，DOM 元素完全不渲染（销毁/重建），有切换成本（项目中弹窗使用 `v-if="show"`，因为弹窗不频繁打开）
- `v-show`：始终渲染 DOM，通过 CSS `display: none` 切换（项目中折叠面板使用 `v-show="!collapsed"`，因为用户频繁切换折叠/展开）
- 选择依据：频繁切换用 `v-show`，不频繁切换用 `v-if`

### Q12: `computed` 和普通函数的区别？

**回答要点：**
- `computed` 会缓存计算结果，依赖数据不变时直接返回缓存值
- 普通函数每次调用都会重新计算
- 项目中 `headerSummaryStr` 使用 computed，因为依赖于 `model.subsidies`，当补助数据变更时自动重新计算
- 方法（带参数的查找）如 getters 中的 `getCompanyNameById(id)` 无法缓存（因为参数变化）

### Q13: BigDecimal 和 double/float 的区别？

**回答要点（金额计算必须使用 BigDecimal）：**
- `float` 和 `double` 是二进制浮点数，无法精确表示十进制小数（如 `0.1` 实际存储为 `0.100000001490116...`）
- `BigDecimal` 使用十进制整数存储，可以精确表示和计算金额
- 项目中所有金额字段（`subsidyTotal`、`allocAmount` 等）都使用 `BigDecimal`
- 比较时使用 `compareTo()` 而非 `equals()`（`equals` 会考虑精度差异，如 `1.0` vs `1.00`）

### Q14: 项目的组件拆分策略是什么？

**回答要点：**
- **通用组件**（`components/common/`）：跨页面复用的基础组件（`SectionHeader`、`FormField`）
- **功能组件**（`components/`）：特定场景使用的组件（`FlatSelect`、`TreeSelect`、`DataTable`、`Pagination`、`SearchBar`、`ConfirmDialog`）
- **详情子组件**（`components/detail/`）：详情页的各个区块（`BasicInfoSection`、`TripSection`、`SubsidyInfoSection`、`CostSummarySection`、`CostAllocationSection`、`RemarkSection`、`ActionBar`）
- **弹窗组件**：行程编辑（`TripDialog`）、补助日历编辑（`SubsidyCalendarDialog`）
- **视图组件**（`views/`）：路由级别的页面组件（`ReimbursementList`、`ReimbursementDetail`）
