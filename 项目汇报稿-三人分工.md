## 成员 A：项目总览与后端核心 + 前端复杂页面（~6 分钟）

> **开场白：面向评委，自信从容，语速适中**

各位老师好，我们是第 X 组，今天我们汇报的项目是**差旅费用报销系统**。

我负责的是项目中最核心的部分——包括后端的核心业务逻辑，以及前端最复杂的两个页面。下面我先介绍整体项目，再展开技术细节。

### 一、技术选型与项目架构（~1.5 分钟）

我们采用**前后端分离架构**。后端用 **Spring Boot 4.1.0** + **MyBatis-Plus 3.5.16**，基于 Java 17；前端用 **Vue 3.5** + **TypeScript 6.0** + **Vite 8.0**。数据库是 **MySQL 8.x**。

后端内部采用经典三层架构：Controller → Service → Mapper。数据传输上做了严格的 DTO/VO/Entity 三层分离——这个是我设计的，前端发来的 JSON 用 DTO 接收，返回给前端的数据用 VO 封装，数据库映射用 Entity。各层职责清晰，不会把数据库字段直接暴露给前端。

这里顺便说一下我们的开发模式——我们三个人都是**前后端都写**的，不是一个人纯后端一个人纯前端。比如我除了后端核心业务，还负责了前端最复杂的那几个页面：报销单详情页和费用分摊区块。这样每个人都能在汇报中展现出全栈能力。

### 二、MyBatis-Plus 深度应用（~2 分钟）

这是我个人投入精力最多的部分。

**第一，零 SQL 实现 CRUD。** 所有 Mapper 接口继承 `BaseMapper<T>`，不用写一行 SQL。Service 继承 `ServiceImpl`，业务代码直接调用 `this.page()`、`this.getById()`、`this.saveOrUpdate()`。

**第二，LambdaQueryWrapper 类型安全的动态查询。** 列表查询是这个系统的核心场景，我在 `FkReimMainServiceImpl.java` 第 72-97 行用 LambdaQueryWrapper 构建动态条件——

```java
LambdaQueryWrapper<FkReimMain> wrapper = new LambdaQueryWrapper<>();
if (StringUtils.hasText(queryDTO.getTitle())) {
    wrapper.like(FkReimMain::getReimbursementTitle, queryDTO.getTitle());
}
wrapper.orderByDesc(FkReimMain::getCreationTime);
```

用 `FkReimMain::getReimbursementTitle` 这种 Lambda 方法引用代替字符串，字段改名编译期就能发现，天然防 SQL 注入。

**第三，分页插件的物理分页。** 我配置了 `PaginationInnerInterceptor(DbType.MYSQL)`，自动在 SQL 末尾拼 `LIMIT`，实现真正的物理分页。

**第四，雪花算法主键。** 所有业务表用 `@TableId(type = IdType.ASSIGN_ID)`，分布式环境下保证全局唯一且趋势递增。

### 三、数据库设计与关键设计决策（~1.5 分钟）

数据库一共 11 张表：6 张基础数据表 + 5 张业务表。

重点讲一个设计决策——**冗余字段策略**。主表 `fk_reim_main` 不仅存了 ID，还同时存了编号和名称。两个原因：查询性能——列表查询不用 JOIN 四五张表；历史数据保护——员工离职后历史报销单仍能显示当时的姓名。这是企业级系统常用的"以空间换时间"策略。

保存时我在 `FkReimMainServiceImpl.java` 第 357-388 行逐个查字典表补全这些冗余字段。

### 四、前端：详情页与费用分摊（~1 分钟）

我除了后端，还写了前端最复杂的两个部分。

**ReimbursementDetail.vue**（520+ 行）是整个前端的核心页面，我实现了 6 个可折叠区块的联动——行程变更后自动重新计算补助金额、更新分摊数据，并且已经保存到数据库的单据自动触发后端保存。

**CostAllocationSection.vue**（550+ 行）实现了费用分摊的完整交互——比例输入自动联动金额、金额输入自动联动比例、均摊功能、以及第一行"剩余补齐"算法保证合计精确等于 100%。

以上就是我负责的部分。接下来由成员 B 介绍前端架构和通用组件。

## 成员 B：后端基础数据 + 前端架构体系（~5 分钟）

> **开场白：承接成员 A，落落大方**

好，谢谢 A 同学的分享。刚才 A 介绍了核心业务逻辑和前端复杂页面，我来介绍我负责的两块——后端的**基础数据平台**和前端的**整个架构体系**。

### 一、后端：基础数据模块（~1.5 分钟）

我负责后端全部 6 张基础数据表的管理——公司、部门、员工、城市、业务类型、项目。这部分看似简单，但有一个技术亮点是**业务类型树形结构的构建**。

数据库中业务类型是平铺存储的，每行记录有一个 `superiorId` 字段指向父节点。我在 `BaseDataServiceImpl.java` 中用 **HashMap + 双循环** 算法把平铺数据组装成树：第一次循环把全部节点放入 HashMap 做索引，第二次循环根据 `superiorId` 把子节点挂到父节点的 `children` 列表下。时间复杂度 O(n)，只遍历两遍。

同时我在 `BaseBusinessType.java` 中用 `@TableField(exist = false)` 标记 `children` 字段——这个字段在数据库表中不存在，只是为了把树形结构传给前端。前端调用 `/base_data/businessTypes` 接口拿到直接可用的树形 JSON，不需要自己做转换。

### 二、前端：架构设计与 Pinia 状态管理（~1.5 分钟）

前端整个架子是我搭的。入口文件 `main.ts` 初始化 Vue + Pinia + Router，`vite.config.ts` 配置 `/api` 代理到后端 8080 端口。

**Pinia 状态管理**是我比较用心设计的地方。`baseDataStore` 有几个设计点：

一是用 `isLoaded` 标记防止重复加载——页面多次进入不会重复请求接口。二是 6 个接口用 `Promise.all` 并发发出——总耗时等于最慢的接口，不是 6 个加起来。三是做了**字段映射**——后端返回的字段名是 `reimCompanyId`、`reimCompanyNo`，我统一映射为前端的 `id`、`no`、`name`，这样所有下拉组件认同一套格式。

还有**递归 Getter**——业务类型是树形结构，我写了一个 `getBusinessTypeNameById`，在树中递归搜索匹配的节点名，组件调用时不需要自己写查找逻辑。

**Pinia reimbursementStore** 封装了所有业务操作——分页列表加载、详情查询、保存草稿、提交、删除、作废，组件只需调用 Store 的方法即可。

### 三、前端：8 个自定义组件（~1.5 分钟）

我们没有用 Element Plus、Ant Design 这些第三方 UI 库，所有组件从零手写。

**TreeSelect 树形选择器**——支持 3 层嵌套渲染，只能选叶子节点（通过 `thereSubordinateNode` 字段判断）。

**DataTable 数据表格**——操作列中有一个"更多"按钮，点击弹出下拉菜单。这里我用了 Vue 3 的 **Teleport** 把菜单渲染到 `<body>` 下，因为如果放在表格内部会被 `overflow: hidden` 裁切。菜单定位用 `getBoundingClientRect()` 计算屏幕坐标，动态设置 `top` 和 `left`。

还有 FlatSelect 下拉选择器、Pagination 分页组件（含省略号算法和快速跳转）、SearchBar 搜索栏、ConfirmDialog 确认弹窗（支持 4 种类型图标）、SectionHeader 折叠区块标题、FormField 表单字段容器——共 8 个。

### 四、Axios 拦截器（~0.5 分钟）

`api/request.ts` 中封装了响应拦截器：后端返回 `{ code, msg, data }` 格式，拦截器先判断 `code !== 200` 就弹窗报错，成功则解包返回 `res.data`。组件拿到的是纯净的业务数据，不需要每次判断 code。

以上就是我负责的部分。接下来由成员 C 来介绍业务核心流程和前后端如何串联。

## 成员 C：后端公共模块 + 前端业务全链路（~5 分钟）

> **开场白：承接成员 B，展现全局视角**

谢谢 B 同学。我负责的部分也是前后端都有——后端方面我写了公共模块和子表服务实现，前端方面我负责了业务工具函数、列表页和详情页的 7 个业务区块。

在项目刚开始的时候，我主要做两件事：一是和 A 一起梳理业务流程，把 12 个 API 的接口契约定下来——每个接口的路径、参数、返回格式都是我先整理的文档；二是把前端需要的三个工具函数先写好——金额格式化、补助计算和表单校验——这样后面开发页面的时候 B 和我都可以直接用。

### 一、后端：公共模块 + 子表服务（~1 分钟）

后端我负责了三块内容。

**第一，统一响应格式 `Result<T>`。** 我定义了包含 `code`、`msg` 和泛型 `data` 的响应结构，前端拦截器就是基于这个结构做的统一解包。

**第二，全局异常处理 `GlobalExceptionHandler`。** 用 `@RestControllerAdvice` 全局拦截 RuntimeException，转成 `Result.error` 返回。关键设计是 HTTP 状态码仍为 200，前端从 `code` 字段统一判断——这样业务异常不会变成前端看不懂的 HTTP 500。

**第三，四个子表的 Service 实现。** 行程、补助、补助日历、费用分摊这些子表的查询和删除逻辑是我写的。比如补助日历的查询需要按日期排序返回，费用分摊需要按 `sortOrder` 排序。

### 二、报销单完整生命周期（~1.5 分钟）

接下来说系统的业务流程——这是我们三个人共同讨论出来的，我负责把完整链路串起来。

报销单有三种状态：草稿、已完成、已作废。

**新建与保存**：用户进入详情页，系统自动初始化空白报销单。填写基础信息、补录行程后，系统自动计算补助——根据到达城市的等级（一线/二线/三线）匹配补助标准，自动生成每天的补助日历。用户可以在日历弹窗中按天勾选或取消各项补助。最后在分摊模块分配费用到各个归属公司和项目。可以随时保存草稿，也可以确认无误后提交。

**提交校验**：提交时触发**前后端双重校验**——我在前端 `validator.ts` 中写了必填校验、同一出行人行程日期不能重叠、分摊比例 = 100%；后端 A 在 Service 层用 `BigDecimal.compareTo()` 再做一遍精确校验。前端校验是提升体验的，后端校验是保底的——前端校验可以被绕过。

**级联删除与作废**：删除报销单时，先删行程、再删补助（内部清理日历）、再删分摊、最后删主表——全部在一个 `@Transactional` 事务中。

### 三、前后端交互核心链路（~1 分钟）

拿提交流程举例——用户点击"确认提交" → 前端 `validator.ts` 校验 → Store 调 API → Axios 发 POST → Vite 代理转发 → 后端 Controller（synchronized 锁）→ Service 校验 + 事务保存 → 返回 Result → Axios 拦截器解包 → 弹窗提示成功。

这条链路上，A 负责后端 Controller/Service，B 负责 Axios/Store，我负责前端校验和全流程串联。

### 四、项目难点与收获总结（~1.5 分钟）

**第一个难点：行程与补助的联动。** 行程变了，补助要跟着变——城市等级决定标准、日期范围决定天数、每天生成日历明细。我把计算逻辑单独抽到 `subsidyCalculator.ts`，前端后端各调用一次，保证一致性。

**第二个难点：费用分摊的剩余补齐。** 多行分摊中第一行的比例自动 = 1.0 - 其余行之和，金额同理。还要处理浮点精度——前端用容差 0.0001 比较、后端用 BigDecimal。

**第三个难点：临时 ID 策略。** 新建记录时还没有数据库 ID，前端用 `NEW_`、`T_NEW_`、`S_NEW_` 等前缀标识临时数据。后端保存时识别这些前缀，用雪花算法生成正式 ID。

**我们的收获**：这个项目最特别的是，我们三个人都做了前后端的工作。我既能讲后端公共组件的设计思路，也能讲前端业务页面的实现细节。这种全栈的参与方式，让我们对整个系统有更完整的理解，而不是各管一块互不了解。

以上是我们组的全部汇报内容，感谢各位老师的聆听，请老师们提问。
