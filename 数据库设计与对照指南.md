# 差旅报销系统 — 数据库设计与前后端数据对照指南

本项目是一个前后端分离的系统，数据库设计需要同时支持**基础主数据（静态下拉框选项）**和**用户填写的业务单据数据**。

根据 [差旅报销单详细设计_整合版.docx](file:///d:/Desktop/java/travel-reimbursement/差旅报销单详细设计_整合版.docx) 的技术规范和表结构定义，现已完成对表名、字段名、数据类型以及前后端映射逻辑的全面对齐与整理。

## 一、 数据库表结构设计（SQL DDL）

你可以直接在 MySQL 中运行以下 SQL 来创建表结构。本设计采用 InnoDB 引擎，并建立了合理的索引与外键约束以保证数据一致性。

```sql
-- 创建并使用数据库
CREATE DATABASE IF NOT EXISTS `travel_reimbursement` DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_general_ci;
USE `travel_reimbursement`;

SET FOREIGN_KEY_CHECKS = 0;

-- ==========================================
-- 1. 基础主数据表（用于下拉菜单展示）
-- ==========================================

-- 1.1 费用归属公司表
CREATE TABLE IF NOT EXISTS `base_company` (
  `reim_company_id` varchar(32) NOT NULL COMMENT '公司唯一标识ID',
  `reim_company_no` varchar(32) NOT NULL COMMENT '公司编号',
  `reim_company_name` varchar(100) NOT NULL COMMENT '公司名称',
  PRIMARY KEY (`reim_company_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='费用归属公司基础表';

-- 1.2 报销部门表
CREATE TABLE IF NOT EXISTS `base_department` (
  `reim_department_id` varchar(32) NOT NULL COMMENT '部门唯一标识ID',
  `reim_department_no` varchar(32) NOT NULL COMMENT '部门编号',
  `reim_department_name` varchar(100) NOT NULL COMMENT '部门名称',
  PRIMARY KEY (`reim_department_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='报销部门基础表';

-- 1.3 员工基础数据表
CREATE TABLE IF NOT EXISTS `base_employee` (
  `reimburser_id` varchar(32) NOT NULL COMMENT '员工唯一标识ID',
  `reimburser_no` varchar(32) NOT NULL COMMENT '员工工号',
  `reimburser_name` varchar(100) NOT NULL COMMENT '员工姓名',
  PRIMARY KEY (`reimburser_id`)

-- 1.4 业务类型表
CREATE TABLE IF NOT EXISTS `base_business_type` (
  `business_type_id` varchar(32) NOT NULL COMMENT '业务类型ID',
  `business_type_no` varchar(32) NOT NULL COMMENT '业务类型编号',
  `business_type_name` varchar(100) NOT NULL COMMENT '业务类型名称',
  `there_subordinate_node` char(1) NOT NULL DEFAULT '0' COMMENT '是否有下级节点 (0:否, 1:是)',
  `superior_id` varchar(32) NOT NULL DEFAULT 'none' COMMENT '上级ID，"none"表示顶级',
  PRIMARY KEY (`business_type_id`)

-- 1.5 城市表

  `city_name` varchar(100) NOT NULL COMMENT '城市名称',
  `city_type` char(1) NOT NULL COMMENT '城市类型 (1:一线, 2:二线, 3:三线)',
  PRIMARY KEY (`city_no`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='出差城市标准基础表';

-- 1.6 项目基础表
CREATE TABLE IF NOT EXISTS `base_project` (

  `project_no` varchar(32) NOT NULL COMMENT '项目编号',
  `project_name` varchar(100) NOT NULL COMMENT '项目名称',

-- 2. 业务单据交易数据表（严格对齐详细设计规范）

-- 2.1 报销单主表
DROP TABLE IF EXISTS `fk_reim_main`;
CREATE TABLE `fk_reim_main` (
  `id` varchar(32) NOT NULL COMMENT '主键ID',
  `reim_no` varchar(32) NOT NULL COMMENT '报销单号(RCBX+日期+序号)',
  `creation_time` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` datetime DEFAULT NULL ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  `reimbursement_title` varchar(500) DEFAULT NULL COMMENT '报销标题',
  `reimburser_id` varchar(32) DEFAULT NULL COMMENT '报销人ID',
  `reimburser_no` varchar(20) DEFAULT NULL COMMENT '报销人工号',
  `reimburser_name` varchar(50) DEFAULT NULL COMMENT '报销人姓名',
  `reim_department_id` varchar(32) DEFAULT NULL COMMENT '报销部门ID',
  `reim_department_no` varchar(20) DEFAULT NULL COMMENT '报销部门编号',
  `reim_department_name` varchar(50) DEFAULT NULL COMMENT '报销部门名称',
  `reim_company_id` varchar(32) DEFAULT NULL COMMENT '费用归属公司ID',
  `reim_company_no` varchar(20) DEFAULT NULL COMMENT '费用归属公司编号',
  `reim_company_name` varchar(50) DEFAULT NULL COMMENT '费用归属公司名称',
  `business_type_id` varchar(32) DEFAULT NULL COMMENT '业务类型ID',
  `business_type_no` varchar(50) DEFAULT NULL COMMENT '业务类型编号',
  `business_type_name` varchar(100) DEFAULT NULL COMMENT '业务类型名称',
  `business_trip_reason` varchar(500) DEFAULT NULL COMMENT '出差事由',
  `subsidy_total` decimal(12,2) DEFAULT '0.00' COMMENT '补助总金额',
  `meal_allowance` decimal(12,2) DEFAULT '0.00' COMMENT '餐费补助',
  `transportation_allowance` decimal(12,2) DEFAULT '0.00' COMMENT '交通补助',
  `phone_allowance` decimal(12,2) DEFAULT '0.00' COMMENT '通讯补助',
  `remarks` varchar(1000) DEFAULT NULL COMMENT '备注信息',
  `status` varchar(2) NOT NULL DEFAULT '0' COMMENT '单据状态 (0:草稿, 1:已完成, 2:已作废)',
  `create_user_id` varchar(32) DEFAULT NULL COMMENT '创建人ID',
  `create_user_name` varchar(50) DEFAULT NULL COMMENT '创建人姓名',
```

## 三、 数据流向对照与持久化逻辑

### 1. 前端传过来的数据是如何落地的？

当用户在前端页面点击 **“保存草稿”** 或 **“提交”** 时，前端会向后端发送一个大的 JSON 包。这个 JSON 包包含基本信息、行程列表、分摊列表等，后端需要在一个本地事务中将数据拆分保存至以下数据库表中：

#### ① 基本信息 (落在 `fk_reim_main` 主表)
*   前端表单字段（`reimburserId`、`reimDepartmentId`、`reimCompanyId`、`businessTypeId` 等）对应主表的同名外键字段。
*   报销单标题对应 `reimbursement_title`，出差事由对应 `business_trip_reason`。
*   后端收到保存请求时，需要通过服务层自动生成一个单号写入 `reim_no`，格式规则为 `RCBX + yyyyMMdd + 4位顺序自增序号`。
*   保存草稿时单据状态 `status = "0"`，提交报销单时单据状态 `status = "1"`。
*   金额合计字段包括 `subsidy_total`（补助总金额）、`meal_allowance`（餐费补助合计）、`transportation_allowance`（交通补助合计）以及 `phone_allowance`（通讯补助合计），均为 `decimal(12,2)` 类型。
*   备注信息字段为 `remarks`（最大1000字符）。

#### ② 补录行程 (落在 `fk_reim_trip` 表)
*   前端行程数组中的每一项行程，对应 `fk_reim_trip` 中的一条行程记录。
*   每笔行程数据包含出行人 `person_id`、工号 `person_no`、姓名 `person_name`，出发城市编号 `departure_city_no`，到达城市编号 `arrival_city_no`，出发日期 `start_date` 和到达日期 `end_date`。
*   插入数据库时，行程表的外键 `reim_id` 指向主表 `fk_reim_main.id`。

#### ③ 补助信息明细 (落在 `fk_reim_subsidy` 表)
*   **后端自动生成逻辑**：保存/修改行程时，后端会根据每一条行程记录，联动生成一条补助信息记录。
*   字段 `subsidy_city` 自动取行程中的到达城市（对应前端 `arrivalCityName`）。
*   日期范围 `start_date` 和 `end_date` 自动对应行程的出发日期和到达日期。
*   补助天数对应 `days` 字段。
*   `apply_amount` 对应申请金额（为补助日历的满额标准金额之和），`subsidy_amount` 对应实际补助金额（即用户自行核减、勾选后计算出的实际金额）。

#### ④ 补助日历 (落在 `fk_reim_subsidy_calendar` 表)
*   **计算标准逻辑**：根据补助信息关联的城市类型，系统会获取该天补助的标准上限：
    *   **一线城市（北京/上海）**：餐费标准 `meal_standard` 为 100 元/天；
    *   **二线城市（武汉/杭州）**：餐费标准 `meal_standard` 为 80 元/天；
    *   **三线城市（荆州等）**：餐费标准 `meal_standard` 为 50 元/天；
    *   **交通补助与通讯补助标准**：分别为 `transport_standard` 与 `phone_standard`，所有城市均为固定 40 元/天。
*   **金额对照**：若前端勾选（如 `mealChecked = true` 对应数据库 `meal_checked = "1"`），则实际金额 `meal_amount` 等于标准金额 `meal_standard`；若未勾选（对应数据库 `meal_checked = "0"`），则实际金额 `meal_amount` 写入 `0.00`。
*   主表中的餐费补助 `meal_allowance`、交通补助 `transportation_allowance` 和通讯补助 `phone_allowance` 则由所有日历明细的实际补助金额汇总累加而来。

#### ⑤ 费用分摊 (落在 `fk_reim_cost_allocation` 表)
*   分摊数组对应报销费用的分摊占比情况。
*   分摊比例对应字段 `ratio`（DECIMAL(6,4) 存储，如 `0.5000` 代表 50%）。
*   分摊金额对应字段 `amount`。
*   外键字段关联对应的 `company_id`、`company_name` 和 `project_id`、`project_name` 等。

## 四、 核心保存与提交逻辑控制（Java 端注意事项）

由于这是一套强关联的数据，保存/提交时后端接口应该在 Service 层使用 **`@Transactional` 事务注解**，按照以下步骤处理持久化：

```
1. 根据前端传过来的实体 ID，清空旧的关联明细数据：
   DELETE FROM fk_reim_trip WHERE reim_id = ?;
   DELETE FROM fk_reim_subsidy WHERE reim_id = ?;
   DELETE FROM fk_reim_cost_allocation WHERE reim_id = ?;
   (注：因为级联外键约束，删除 fk_reim_subsidy 会自动删除关联的 fk_reim_subsidy_calendar 明细)

2. 保存或更新报销单主表信息 (fk_reim_main)
3. 批量保存最新的行程数据 (fk_reim_trip)
4. 根据行程自动生成并批量保存补助信息与补助日历明细 (fk_reim_subsidy & fk_reim_subsidy_calendar)
5. 批量保存分摊比例数据 (fk_reim_cost_allocation)
```
